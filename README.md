# PV–PP Grenade Self-Sacrifice Benchmark

## One-Click Runner

For a simple public-facing demo run, the repository root includes `runit.command`. This only runs a sample benchmark of 100 mixed iterations and is intended only to demonstrate the benchmark features. It does not run the individual scenarios or large benchmark batches with a significant number of iterations.

On macOS, you can make it executable once with:

```bash
chmod +x runit.command
./runit.command
```

This launcher runs a 100-run mixed benchmark, creates the output directories on first use, audits the generated replay set, and then opens the replay viewer against `data/grenade_missionflow_replays_v3`.

If you prefer to run the tools manually or use different scenarios, seeds, or run counts, use the commands shown below in Quick Start.

## Quick Start

Run the benchmark directly with `src/benchmark.py`.

Example:

```bash
python3 src/benchmark.py --max-ticks 56 --map-mode varied --write-replay-trace --runs 100 --seed 1999 --scenario extraction_moderate
```

Other common workloads:

```bash
python3 src/benchmark.py --max-ticks 56 --map-mode varied --write-replay-trace --runs 100 --seed 999 --scenario extraction_easy
python3 src/benchmark.py --max-ticks 56 --map-mode varied --write-replay-trace --runs 100 --seed 2999 --scenario extraction_difficult
python3 src/benchmark.py --max-ticks 56 --map-mode varied --write-replay-trace --runs 100 --seed 3999 --scenario extraction_improbable
```

The current default run count in `src/benchmark.py` is **100** unless you override it with `--runs`.

This repository does **not** ship a prebuilt `data/` directory. The `data/` directory is created when you run the benchmark for the first time.

To inspect replay traces generated by the current benchmark after a run:

```bash
python3 src/replay_viewer.py --dir data/grenade_missionflow_replays_v3
```

To audit replay traces generated by the current benchmark after a run:

```bash
python3 src/replay_trace_audit.py --dir data/grenade_missionflow_replays_v3 --print --limit 100
```

If you choose to generate or curate a different replay set locally, point the viewer and audit tool at that directory instead.

## Repository Overview

This repository has two linked purposes:

1. provide the grenade self-sacrifice benchmark, and
2. provide a substantial working release set for the PV–PP framework that the benchmark instantiates.

The benchmark is therefore not presented as an isolated simulation trick. It is presented as one benchmark implementation inside a broader non-scalar framework of viability, action, and decision.

At the benchmark level, the central question is narrow:

Can a non-scalar PV–PP implementation and a self-preserving scalar comparator behave similarly in easier rescue cases, yet diverge in grenade-bearing extraction cases where direct sacrificial protection becomes decision-relevant?

## Repository Purpose

This repository is intended to support:

## Benchmark Question

The benchmark asks:

In a matched rescue-and-extraction environment, do PV–PP and a self-preserving scalar comparator coincide in easier cases, and does PV–PP achieve higher recoveree extraction rates in grenade-bearing cases where direct sacrificial protection becomes structurally relevant?

A narrower formulation of the same question is:

Can sacrificial blast absorption become selectable through governing structure, feasibility, adequacy, and selection without requiring scalar aggregation across persons as the governing decision rule?

## Comparator Discipline

The benchmark compares two decision architectures operating in the same simulated environment:

- `pvpp`
- `baseline_self_preserving`

Both operate under the same map, terrain, cover, background-fire process, grenade process, objective geometry, extraction geometry, movement constraints, and burden mechanics. The comparison is therefore architectural, not environmental.

The comparator is a self-preserving scalar baseline. That matters. This repository does **not** claim that scalar models in general cannot represent sacrifice. A scalar model can always be written with weights, priorities, or team-level utility structure that make sacrificial action utility-maximizing. The narrower point is that PV–PP can generate the benchmark rescue-and-sacrifice pattern without scalar aggregation as the governing decision rule, and that it outperforms this particular self-preserving scalar comparator in the benchmark’s harder cases.

## Headline Result

The current benchmark supports a narrow pattern claim:

- PV–PP and the scalar self-preserving baseline converge in easy cases.
- PV–PP achieves materially higher recoveree extraction rates in the moderate and difficult grenade-bearing workloads.
- The PV–PP advantage persists in an improbable tail case, though with reduced magnitude.
- A mixed benchmark also preserves a meaningful PV–PP extraction advantage.

The intended empirical shape is therefore:

- parity at the easy edge
- strongest divergence in the middle tiers
- partial reconvergence in near-hopeless conditions

## Current Benchmark Structure

The benchmark uses:

- a 12×12 grid
- a five-soldier squad
- an elevated objective zone
- a home-side extraction zone
- terrain height and cover
- background enemy fire
- wounding and mobility degradation
- burdened withdrawal dynamics
- grenade events during withdrawal

The benchmark compares the same environment under two different decision architectures. It is not a comparison between two different maps, two different threat models, or two different movement engines.

## Main Workloads

The current benchmark ladder uses:

- `extraction_moderate`
- `extraction_difficult`
- `extraction_improbable`
- `mixed`

The ordered ladder is the main benchmark. The mixed workload is supplementary.

## How to Read the Results

Read the benchmark in this order:

1. `extraction_easy` as a convergence/control case.
2. `extraction_moderate` and `extraction_difficult` as the strongest evidence of architectural divergence.
3. `extraction_improbable` as a tail stress case.
4. `mixed` as a supplementary robustness check rather than the main ordered benchmark.

Also read the casualty tradeoff directly rather than smoothing it over: PV–PP’s extraction advantage is not free. In the stronger divergence workloads, PV–PP generally accepts slightly higher average total casualties in exchange for higher recoveree extraction rates.

One additional caveat matters: the current published result set uses the same seed for `extraction_improbable` and `mixed`. That does not erase the pattern, but it is a real statistical packaging weakness and should be stated as such rather than hidden.

## What This Repository Does Not Claim

This repository does **not** claim:

- that scalar models in general cannot represent self-sacrifice
- that PV–PP has already been shown to reduce to scalar aggregation in all scalar-valid environments
- that this benchmark proves the full PV–PP theory
- that PV–PP universally dominates all scalar formulations
- that PV–PP dominates the scalar comparator on every metric
- that the benchmark fully isolates every source of divergence beyond the center-square sacrificial case

The benchmark claim is narrower than that. The broader framework release also should not be confused with closure of every separate theoretical workstream.
